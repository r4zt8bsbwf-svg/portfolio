SubSpectra

An autonomous water tank inspection ROV — bringing live video, environmental data, and precise control into confined submerged environments.

Role Systems integration, firmware, UI
Stack Raspberry Pi · Arduino · Python · OpenCV · PySide6
Type Capstone / Long-form
Status Completed

The problem

Inspecting the inside of municipal and industrial water tanks usually means sending divers into confined, dark, sometimes hazardous spaces. SubSpectra was built to make that unnecessary — a remotely operated vehicle that gives operators live video, environmental data, and fine maneuverability without anyone ever getting wet.

The hard part wasn't any single subsystem. It was making real-time motor control, networking, computer vision, sensor calibration, and waterproofing all cooperate in one chassis.

Architecture

The compute is intentionally split across two boards, each doing what it's good at:

Raspberry Pi 5 — high-level

  • Camera streaming and OpenCV processing pipeline
  • Ethernet communication with the surface UI
  • Sensor aggregation, telemetry processing, logging
  • High-level control logic and mode management

Arduino Mega — real-time

  • ESC signal generation and thruster PWM output
  • Watchdog safety systems
  • Lower-level sensor polling

The two communicate over USB serial using structured JSON packets. The split exists because Linux on the Pi is not deterministic and motor control needs precise timing — if the Pi lags or crashes, the Arduino keeps PWM stable.

Propulsion

Eight thrusters: four large units at roughly 30° for translation and yaw, four vertical units at the corners for depth, stabilization, and roll/pitch correction. Driven through ESCs with servo-style PWM around 1500 μs neutral, 1900 μs forward, 1100 μs reverse. The control layout means the large motors handle operator-driven movement while the smaller verticals auto-level the vehicle in the background.

Sensors

  • MS5837-30BA for pressure and depth, zeroed at the surface
  • DS18B20 waterproof temperature sensor over OneWire
  • DFRobot pH, TDS, and conductivity sensors — each with their own calibration routines, reference solutions, and noise filtering

Vision

An OAK-D Lite stereo camera streams live video to the operator UI over Ethernet, with an OpenCV processing pipeline for future computer vision expansion. Getting this stable meant working through USB bandwidth limitations, Pi undervoltage events under camera load, autofocus quirks, frame latency, and compression tradeoffs.

Operator UI

The custom interface is built in PySide6 with PyQtGraph OpenGL for the 3D vehicle visualization. It shows live video, sensor overlays, motor outputs, latency, and operating mode (manual, stabilize, angle hold, inspection). A simulation mode lets the UI run without hardware connected — useful for iterating on layout without setting up the full rig.

Safety & reliability

  • Watchdog timeout — thrusters stop automatically if comms drop
  • Line fuses and main fuse protection, tested under load
  • Telemetry validation flags missing sensors, invalid readings, and out-of-range values

Mechanical

Waterproof PVC enclosure, acrylic viewing window, waterproof cable glands, externally mounted ESCs, PETG structural components, 3D-printed mounts. A lot of the fabrication time went into epoxy sealing, pressure resistance, waterproof connector consolidation, and routing cables so heat could still escape.

What this project taught me

Most of the real work was at the seams — Pi power instability under camera load, ESC calibration, Ethernet latency, sensor calibration drift, IMU stabilization tuning, PWM debugging with an oscilloscope. Each problem touched two or three disciplines at once. The biggest takeaway: if you can't reason about hardware, firmware, networking, mechanical, and power as one system, you can't ship a vehicle that works underwater.
